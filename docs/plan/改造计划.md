# CC Switch 改造计划

## 核心思路

**Provider 只管 API 接入，Profile 只管"平台选模型"。**
模型粒度配置（上下文、推理、图片等）统一在 Provider 管理中设置。

---

## 阶段 1：后端 — 数据模型 + 路由

### 1.1 Profile 数据模型（db.ts）

**现状：** Profile 绑定单个 app_type + provider_id

**目标：** Profile 支持多平台 target

```ts
interface ProfileTarget {
  id: string
  profile_id: string
  app_type: string       // 'codex' | 'claude' | 'gemini' | 'opencode' | 'workbuddy'
  model: string           // 模型名
  // Claude 专用
  claude_haiku?: string
  claude_sonnet?: string
  claude_opus?: string
  // Codex 专用
  codex_models?: string[]  // 模型列表（多选）
}

interface Profile {
  id: string
  name: string
  kind: 'single' | 'bundle'
  slug: string
  extra_config: string    // 保留，存其他扩展数据
  created_at: string
  updated_at: string
  targets: ProfileTarget[]
}
```

- `Profile` 去掉 `app_type` 和 `provider_id`
- 新增 `kind` 字段和 `profile_targets` 数组
- `target.model` 从所有 Provider 的模型池中选（不绑定单个 Provider）

### 1.2 Provider 数据模型（db.ts）

**Provider.extra_config 调整：**

```ts
extra_config: {
  provider_type?: string
  capabilities?: Partial<Record<ApiFormat, boolean>>
  models?: Array<{
    id: string                    // 模型 ID
    displayName?: string
    // 通用模型参数
    contextWindow?: number        // 上下文窗口
    maxOutputTokens?: number      // 最大输出
    supportsImages?: boolean      // 是否支持图片
    supportsParallelToolCalls?: boolean
    // 推理参数
    supportsReasoning?: boolean
    defaultReasoningEffort?: string  // low | medium | high | xhigh
    supportedReasoningEfforts?: string[]
    // 输入模态
    inputModalities?: string[]    // ['text'] | ['text', 'image']
    // 系统指令
    baseInstructions?: string
    // WorkBuddy 专用
    supportsToolCall?: boolean
    useCustomProtocol?: boolean
  }>
  // 移除 claude 和 codex 顶级字段（模型映射移到 models 数组里）
}
```

### 1.3 Provider 路由

**新增：**
- `GET /api/providers/:id/models/manage` — 获取该 Provider 已保存的模型列表
- `PUT /api/providers/:id/models` — 保存模型选择（全量覆盖），包含每个模型的粒度参数
- `GET /api/providers/models/all` — 获取所有 Provider 的所有模型（供 Profile 页下拉用，带分组）

**保留：**
- CRUD、获取模型列表（`GET /models` 调 API）、测试连通性、API Key 掩码

**清理：**
- Provider 保存/更新时不再接受 Claude 模型映射和 Codex 目录顶级字段
- 这些数据统一走 `extra_config.models` 数组

### 1.4 Profile 路由

**改造：**
- `GET /api/profiles` — 返回 Profile 列表 + 展开 targets
- `POST /api/profiles` — 创建，接收 `{ name, kind, targets: [{ app_type, model, ... }] }`
- `PUT /api/profiles/:id` — 编辑
- `DELETE /api/profiles/:id` — 删除 Profile + targets
- `POST /api/profiles/:id/apply` — 应用 Profile：
  - 遍历所有 targets
  - Codex: 写 `model_catalog.json` + `config.toml`
  - Claude: 写 `settings.json`（ANTHROPIC_MODEL 等环境变量）
  - Gemini: 写 `.env` + `settings.json`
  - OpenCode: 写 `opencode.json`
  - WorkBuddy: 写 `models.json`（从 Provider 的 models 数据生成）

### 1.5 概览状态路由

- `GET /api/status` — 返回每个平台当前应用的 Profile（不再只是 Provider）
- 新增 `GET /api/overview/profiles/:appType` — 获取某平台可用的 Profile 列表

### 1.6 旧数据迁移

启动时检测旧 Profile（有 `app_type`/`provider_id`），自动迁移：
- 旧 Profile → 新 Profile（kind='single'）+ 1 个 ProfileTarget
