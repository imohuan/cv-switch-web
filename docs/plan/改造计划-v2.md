> **版本说明**：v2 基于 2026-07-10 代码实际状态重新构建，废弃旧版计划中"重新设计数据结构"的路径，改为在当前代码基础上渐进改造。

## 一、当前代码真实状态

### 1.1 数据模型（`backend/src/db.ts`）

| 概念 | 当前结构 | 关键字段 |
|------|---------|---------|
| Provider | 扁平结构 | `id`, `name`, `base_url`, `api_key`, `model`, `api_format`, `extra_config`(JSON string) |
| Profile | 绑定单 Provider + 单 app_type | `id`, `name`, `app_type`, `provider_id`, `slug`, `extra_config` |
| AppStatus | 一对一映射 | `app_type`, `current_provider_id` |

**结论：没有 `ProfileTarget` 表、没有 `kind` 字段。Profile 仍然是 `(app_type, provider_id)` 二元组。**

### 1.2 extra_config 结构（`backend/src/services/providerConfig.ts`）

```ts
interface ProviderExtraConfig {
  provider_type?: string          // 'custom' | 'newapi'
  capabilities?: Record<ApiFormat, boolean>
  claude?: {
    defaultModel?, haikuModel?, sonnetModel?, opusModel?, smallFastModel?
  }
  codex?: {
    defaultModel?: string
    models?: Array<{ model, displayName?, contextWindow?, ... }>
  }
}
```

**结论：Claude 模型映射和 Codex 模型列表仍然在 `extra_config` 的顶级字段里。旧计划说的"移到 models 数组"完全没做。**

## 二、v2 改造路线

核心思路不变：**Provider 只管 API 接入，Profile 只管"平台选模型"。**
但实施路径从"推倒重来"改为"在现有代码上渐进改造"。

### 阶段 A：后端 — 补齐 Provider 模型管理 API

> 优先级：最高。前端 ModelSettings 已经做好，但后端没有对应的持久化 API。

#### A.1 Provider 模型保存 API

**新增：** `PUT /api/providers/:id/models`

请求体：
```json
{
  "models": [
    {
      "id": "gpt-4o",
      "displayName": "GPT-4o",
      "contextWindow": 128000,
      "maxOutputTokens": 16384,
      "supportsImages": true,
      "supportsParallelToolCalls": true,
      "supportsReasoning": true,
      "defaultReasoningEffort": "medium",
      "supportedReasoningEfforts": ["low", "medium", "high"],
      "inputModalities": ["text", "image"],
      "baseInstructions": "You are a helpful assistant."
    }
  ]
}
```

**逻辑：** 全量覆盖写入 `extra_config.models` 数组。同时保留 `extra_config.codex.models` 和 `extra_config.claude` 的向后兼容（不删除旧字段，只是新增 models 数组）。

**为什么不全量迁移？** 旧字段不影响功能，删了反而可能出 bug。新字段和旧字段并存，写入时两边都更新。

#### A.2 Provider 模型读取 API

**新增：** `GET /api/providers/:id/models/manage`

返回 `extra_config.models` 数组（带粒度参数）。

#### A.3 所有 Provider 模型汇总 API

**新增：** `GET /api/providers/models/all`

返回所有 Provider 的模型列表，按 provider 分组：
```json
{
  "providers": [
    {
      "id": "xxx",
      "name": "My New API",
      "models": [{ "id": "gpt-4o", "displayName": "GPT-4o" }]
    }
  ]
}
```

用途：Profile 页的模型下拉数据源。

---

### 阶段 B：后端 — Profile 支持多平台

> 优先级：高。这是改造的核心目标。

#### B.1 Profile 数据模型扩展

**不改 `db.ts` 的 Profile interface。** 而是用 `extra_config` 字段承载多平台数据：

```json
// Profile.extra_config 新结构
{
  "kind": "single",          // 向后兼容：旧 Profile 默认 single
  "targets": [
    {
      "app_type": "codex",
      "model": "gpt-4o",
      "codex_models": ["gpt-4o", "gpt-4o-mini"]
    },
    {
      "app_type": "claude",
      "model": "claude-sonnet-5",
      "claude_haiku": "claude-haiku-4-5",
      "claude_sonnet": "claude-sonnet-5",
      "claude_opus": "claude-opus-4-8"
    }
  ]
}
```

**为什么用 extra_config 而不是新表？**
- 不需要数据库迁移
- 旧 Profile 仍然能正常工作
- 前端读取 extra_config 判断是新格式还是旧格式

#### B.2 Profile 路由改造

| 路由 | 改动 |
|------|------|
| `GET /api/profiles` | 不变，但返回数据中 extra_config 包含 targets |
| `POST /api/profiles` | 接收新格式：`{ name, kind, targets }`，写入 extra_config。向后兼容旧格式 `{ name, app_type, provider_id }` |
| `PUT /api/profiles/:id` | 新增：编辑 Profile 名称和 targets |
| `DELETE /api/profiles/:id` | 不变 |
| `POST /api/profiles/:id/apply` | 改为遍历 targets，每个 target 按 app_type 写配置 |

#### B.3 apply 逻辑改造

当前 `writeProfileConfig` 依赖 `profile.provider_id` 找到 Provider。改造后需要支持"每个 target 独立指定模型，从 Provider 模型池中查找"。

**方案：** apply 时，每个 target 的 `model` 字段去所有 Provider 的 `extra_config.models` 数组中匹配，找到对应的 Provider 和模型参数，然后写配置。

如果匹配不到，降级为：用第一个支持该 app_type 的 Provider 的 base_url 和 api_key。

---

### 阶段 C：前端 — ProfileTab 接入真实 API

> 优先级：高。前端 UI 框架已搭好，就差接入 API。

#### C.1 改造 `useProfileForm.ts`

当前 `useProfileForm` 是旧版（单 app_type + 单 provider_id）。需要：
- 创建/编辑 Profile 时构造 `targets` 数组
- 调用 `POST /api/profiles` 或 `PUT /api/profiles/:id`
- 模型下拉数据源改为 `GET /api/providers/models/all`

#### C.2 ProfileTab.vue 数据接入

- 列表加载从 `GET /api/profiles` 读取，解析 `extra_config.targets`
- 每个 Profile 卡片显示平台摘要（遍历 targets）
- 编辑 Dialog 回填已有 targets 数据

#### C.3 去掉旧字段依赖

Profile 卡片/列表中不再显示 `provider_name`（因为现在一个 Profile 可能对应多个 Provider）。

---

### 阶段 D：前端 — 概览页改为 Profile 驱动

> 优先级：中。当前概览页是 Provider 直连模式，改为 Profile 模式。

#### D.1 StatusOverview 改造

当前逻辑：每个平台 Tab → 下拉选择 Provider → 调用 switch API。

改造后：
- 显示当前平台应用的 Profile（从 `GET /status` 获取）
- 下拉改为 Profile 列表（筛选该平台有 target 的 Profile）
- 切换时调用 `POST /api/profiles/:id/apply`
- Provider 直连切换仍然保留作为 fallback

#### D.2 状态 API 扩展

`GET /api/status` 返回数据中增加 `current_profile_id` 和 `current_profile_name`。

---

### 阶段 E：收尾清理

#### E.1 可选清理

- WorkBuddyModels.vue：保留。它是独立功能，和 Profile 体系不冲突
- 旧版 `useProfileForm.ts`：重构后删除旧逻辑
- Provider 卡片中的 Claude/Codex 模型字段：保留（向后兼容）

#### E.2 不做的事

- ❌ 不删除 `extra_config.claude` 和 `extra_config.codex` 旧字段
- ❌ 不新建 `profile_targets` 表
- ❌ 不改变 `db.ts` 的 Profile interface
- ❌ 不做自动数据迁移（旧 Profile 继续正常工作）

## 四、与旧版计划的关键差异

| 旧版计划 | v2 计划 | 原因 |
|---------|---------|------|
| 新建 ProfileTarget 表 | 用 extra_config JSON 字段 | 避免数据库迁移，减少风险 |
| 删除 Claude/Codex 旧字段 | 保留，新增 models 数组并存 | 向后兼容，不破坏现有功能 |
| 前端重写 ProfileTab | 在现有 UI 框架上接入 API | UI 已经做好了，只需接数据 |
| Provider 模型粒度参数必须配置 | 可选，默认值自动填充 | 降低使用门槛 |
| 删 WorkBuddyModels.vue | 保留 | 它是独立功能，删了没收益 |
| 一次性全量改造 | 渐进式，每阶段可独立交付 | 降低风险，每阶段可测试 |
